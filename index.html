title: Java还是Python，Python还是Ruby，这是个问题。
date: 2016-08-03

tags:

《Java编程思想》的第一章《对象导论》、第二章《一切都是对象》硬啃完了，然后怀着很大的热情用一个星期搭建好了博客，又花3天时间添加友情链接、评论、调整主题等。想要再回去学习时发现对Java的兴趣已经不能往前迈动一步了。

可能由于周日听了一个分享，对写作的思路有个重大突破，注意力都转移到写作上了。对于编程学习，精力有限，选择一定要谨慎。在看了《穷查理宝典》的序言后，我又学会了一招：查别人学习失败的例子，看别人是由于什么放弃的，以免步其后尘。于是搜索了：

python+放弃
ruby+放弃
python+ruby
快速学会java写脚本

我搜到了一些资源：

田腾飞的博客 codertian
Java学习之路：不走弯路，就是捷径。zy950 51CTO
为什么现在总有人劝我放弃java 开源中国社区
写给新手程序员的一封信

不打算附链接了，需要的自己去搜。

Java还是Python

总之看了大概两个半天之后，总结是：学Java需要的时间较长，而我只需要学会写测试脚本就可以了，并期望在一两个月内就能够自己动手写脚本；所以我可以选择比较容易上手、现在也很流行的Python或者Ruby。另外再想想，Vamei好像也是先Python后Java。

Python还是Ruby

对于这两个选择我又犯难了，Ruby网上有很多人表示喜欢，赞其优美、有魔法、社区有活力，而我也有学习Ruby的强大资源。可是我对于去年在Mooc上学到一半的Python课程一直念念不忘，一直希望有时间能够接续下去。在网上搜索了“python+ruby”比较两者之后，主要看了两个文章：

再谈PHP、Python与Ruby
http://www.nowamagic.net/librarys/veda/detail/2504
新手选择 Ruby 还是 Python 学习？
https://ruby-china.org/topics/28368

发现Python除了我喜欢的直接高效的特性外，还有一个Ruby没有的特性：

可以做数据分析

虽然作者也说了：

Python的数据分析能力并不是那么好

但是我认为其数据分析能力一定可以满足我所编写脚本的要求。对方很可能是以较高的标准来判定的，毕竟能写出这篇文章的人，开发水平一定不低，高手自然高标准么。而我在编程能力方面与人竞争肯定没有优势。我的优势在于分析能力。

所以，既然我对Python有兴趣、它又容易上手、并且还有我感兴趣的独有特性：数据分析，那就选它了吧！至于Java，我打算先把同事用Java写的脚本，先抄一遍背一遍，应该能有一个大概的感觉了，以后有机会再慢慢学，Java不放弃，也不着急学，从Python入手。一周至少腾出半天时间来专门学习去年中断的课程。

我遇到的坑

搭建博客路上遇到的坑有大有小，小坑就是至少一小时内能解决的坑，大坑就是上面花了大部分时间才弄清楚的坑。下面介绍这些天我遇到的6个小坑1个大坑。

小坑

小坑1:

配置开发环境过程中，出现提示：

-bash: /Users/myname/.bashprofile: Permission denied

Google以上提示，查到网页：
https://github.com/rvm/rvm/issues/2832
找到解决方案：

nabins13 commented on Dec 1, 2015
i just did was:
sudo nano ~/.bash_profile
and added the line ‘source ~/.profile

it workes fine

但出现新的问题：修改后如何退出？如何保存文件？由此引出小坑2（我对linux的操作一窍不通）。

小坑2:

Google关键字“sudo nano 怎么保存文件”，查到网页：
http://429006.com/article/technology/1804.htm
找到解决方案：

完成后按ctrl+x退出，按y保存，回车，OK

小坑3:

配置开发环境过程中，出现提示：

-bash: nvm: command not found

Google以上提示，查到网页：
http://askubuntu.com/questions/269631/nvm-command-not-found/311332
找到解决方案：

The problem is that you need nvm to be automatically sourced upon login, so either add the following line to your ~/.bashrc or ~/.profile file.

[[ -s $HOME/.nvm/nvm.sh ]] && . $HOME/.nvm/nvm.sh
这个问题也可能是按照当时本地计算机里的提示操作的，具体已经忘记了。不过由于之前已经学会了如何打开、编辑、保存、退出文件，这个问题相对小坑1已经变得简单多了。

小坑4:

搭建博客后期，输入hexo deploy指令后，出现提示：

fatal: Not a git repository (or any of the parent directories): .git

Google以上提示，查到网页：
http://stackoverflow.com/questions/20413459/fatal-not-a-git-repository-or-any-of-the-parent-directories-git
找到解决方案：

Are you in the right directory? Does typing ls show the right files?
Have you initialized the repository yet? Typed git init
小坑5:
username要改为自己的用户名，不多说了。这个坑必有人躺。

小坑6:

新建的repo名字：yourname.github.io，yourname一定要用自己的用户名，这个也不多说了。此坑也必有人跳。github上可以修改用户名，repo名称，具体方法自己找，网上很多资源。repo就是repository的缩写。

大坑

小坑很多，但找到解决方案后，很快就能解决。最后一个大坑：

open yourname.github.io
稍等一下，你就可以看到默认浏览器被打开，且已经浏览到 yourname.github.io 这个网址了……

这是我花了大量时间解决，到现在也没解决，最后发现其实无关紧要的小问题。

先说明下，我其实对整个流程并不懂，只是按照步骤顺序操作，有问题就Google，解决了再下一步。所以当时我压根不懂hexo deploy这一步成功之后，即出现：

INFO Deploy done: git

其实博客就已经搭建成功了。但当时无知的我认为：

hexo deploy
open yourname.github.io

是紧密联系在一起的，一定要在命令行输入open yourname.github.io自动打开浏览器，搭建博客才算真正成功。（在实际操作中，“yourname”已经改为我的用户名了。）

于是我反复按照教程从头开始操作，重复多次，都没能实现用命令行打开网页。折腾过程中hexo deploy已经成功过很多次了，也就是说我已经搭建成功很多次了只是自己不知道而已。我的关注点一直在用open yourname.github.io指令无缝连接自动打开网页。

我跳出大坑的过程，基本就是弄明白hexo deploy意思的过程。在弄明白之前：

  • 试过改github上的用户名，将大写改为小写，重复教程步骤；
  • 改repo的名字，删除原来的repo重新新建，重复教程步骤；
  • 删除nvm，重新装，重复教程步骤；
  • 删除本地计算机上的yourname.github.io文件夹，重复教程步骤；
  • 调整教程中的指令顺序。
    这些方法都没用。逐渐开始心急如焚、某些脑细胞已经开始骂自己笨了，怎么别人花了一两天就搭建好了，我怎么就弄不好？预计一周完成的，现在快周末了，怎么办？

先告诉自己：耐心，稳定情绪。
再鼓励自己：要有信心，别人不是神，别人能办到的，我也能办到。
接着再想办法：

从前看过一个故事，大意是一束激光割肿瘤，割不掉，因为强度不够；于是人们想了个办法：用多束激光，360度管齐下，肿瘤就给割掉了。

既然一条道走不通，那就看看其他的路呗，试下360度聚焦。网上还有很多其他的教程，看看他们怎么写的，也许能弄明白这教程里些步骤都是啥意思，也许会有转机。

也想过求助他人，但最后仍然觉得请教太耽误双方的时间了，最终决定由自己解决。

然后我Google了关键字“github＋hexo＋搭建博客”，哗哗哗搜到了一大片，看了大概十几篇，选了其中一个跟目前方法最接近、网页显示风格也最喜欢的教程来看：

http://sunwhut.com/2015/10/30/buildBlog/

看了许多教程之后，终于弄明白，hexo deploy这一步成功之后，就表示个人博客已经成功部署到github上了。我就不再纠结用命令行不能打开博客的问题了，直接跳过主教程这一步，往下走。

后面的步骤就相当流畅了，尤其是，走到：

输入 hgs 就是生成内容之后在 localhost:4000 里预览，输入 hgd 就是部署到服务器上……
这一步时，心里直哈哈哈大笑了，成就感满满。当晚睡眠质量从前一晚的65%提升到84%。

总结

坚守目标很重要

我在整个过程中，牢记目标是搭建个人博客，而不是学习atom、markdown、linux指令，只有在需要借助这些工具时，才去学习必要掌握的知识，所以我跳过了教程的很多地方。不过这个方法很适合我这样心急的人，不一定适合其他人。如果有时间，一步步跟着教程走或许也不错。

初步目标达成了，后面还有很多事情要做，比如加评论、统计功能，再学下如何加图片，更好的编辑显示效果。罗马不是一天建成的，泰山不是一步登上去的，慢慢来。

用Java写脚本的机会来了
发表于 2016-08-04 |
今天老大突然问：

你现在会写脚本吗？

我略加思考，再略带心机的回答：

不懂的太多，貌似现在写不了。

先把话堵死，让对方期望值降到谷底。

停顿几秒，接着说：

不过可以试试。

再给对方一丝渺茫的希望。其实昨天老大已经让另一个同事写脚本了。那位同事特意过来问我会不会写，我说不会。所以，我很清楚，老大今天问我，必然是没有其他选择、不得已而问之。

老大于是说：

那你写个脚本，控制下＊＊设备，看能否实现。

窃喜，答：

我试试。

之前看脚本，老有一种弃公营私的感觉，因为这是个人想学习，工作上并未要求。现在，终于可以光明正大地开始看了。

其实我对写脚本毫无把握，因为看别人的脚本，我还是一头雾水，看天书一般。但老大今天递出橄榄枝，我觉得这是个非常好也难得的机会，先抓住再说。因为这样，遇到问题就有人指导了。

前两天已经迷迷糊糊看了两天。今天仍然感到脑子一团麻，无处下手。因为知道了潜意识是大脑的自动化处理脚本，我决定继续昨天的方法，运用潜意识：

一个字母一个字母的抄写现成的脚本，同时一边听与脚本毫无关系的讲座音频。

然而，依然一头雾水，毫无方向。下午五点钟，大脑卡壳了，没法思考了，于是出去逛了十几分钟。回去后又接着看脚本大概十几分钟，看它的运行流程是怎样的。

奇迹竟然出现了。我忽然就看懂了大概，明白该如何实现新的功能了。思路有了，豁然开朗了。其实就是把现有脚本中某个类里的已有属性全部替换成＊＊设备的属性就可以了！明天去试试。
